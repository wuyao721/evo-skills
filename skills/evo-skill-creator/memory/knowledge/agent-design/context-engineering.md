# 上下文工程（Context Engineering）

> 学习日期：2026-03-28
> 来源：多源综合研究（Google Developers Blog、Medium、Anthropic、Elastic 等）
> 学习次数：1

## 核心定义

**上下文工程**是 2026 年取代传统"提示工程"的系统级学科。

- **Prompt Engineering**（微观）：优化"怎么说"——措辞、格式、few-shot 示例
- **Context Engineering**（宏观）：优化"模型看到什么"——数据流、记忆架构、工具路由、状态管理

关键认知：提示本身通常只占上下文窗口的 ~5%。其余 95%（检索文档、工具定义、历史对话、系统状态）才是系统成败的关键。

## 四大框架支柱

### 1. 选择与裁剪（Selection & Pruning）

- **目标**：保持上下文精简
- **策略**：
  - 总结长历史
  - 裁剪无关的工具调用结果
  - 移除过时的对话轮次
  - 滑动窗口缓冲区

### 2. 检索增强上下文（Retrieval-Augmented Context）

- **目标**：动态即时注入
- **策略**：
  - 使用向量数据库或知识图谱注入*仅与当前子任务相关*的事实/文档/记忆
  - 从"推送式 RAG"转向"拉取式 RAG"（agent 自主决定何时搜索什么）

### 3. 上下文隔离（Context Isolation）

- **目标**：防止"上下文污染"
- **策略**：
  - 将单体 agent 拆分为更小的专用子 agent
  - 每个子 agent 维护独立的上下文窗口
  - 仅传递必要的共享状态（全局变量、高层目标）

### 4. 卸载（Offloading）

- **目标**：将"重数据"保留在模型外部
- **策略**：
  - 不要将大型 JSON/文件直接倒入 prompt
  - 给 agent 提供查询特定字段的工具
  - 通过非 LLM 管道预处理数据后再注入摘要

## 分层架构（Progressive Disclosure）

将上下文组织为分层结构，仅在需要时加载：

| 层级 | 内容 | 注意力权重 |
|------|------|-----------|
| 系统层 | agent 身份、基本规则（极简） | 高（开头） |
| 任务层 | 当前会话/步骤的目标 | 高（开头/结尾） |
| 环境层 | 当前需要的工具和文件访问 | 中 |
| 记忆层 | 近期对话（原文）+ 压缩的历史摘要 | 中-低 |

**"笨区"防御**（Dumb Zone Defense）：关键规则、约束和当前任务目标放在上下文的**开头和结尾**，中间放数据密集型信息。

## 记忆分层模型（Tiered Memory）

| 层级 | 内容 | 策略 |
|------|------|------|
| 短期记忆 | 最近 5-10 轮对话 | 原文保留，保持局部连贯性 |
| 情景记忆 | 之前任务"片段"的总结版本 | 任务导向的密集摘要 |
| 语义记忆 | 外部向量存储 | 仅按需访问 |

## 压缩策略

### 1. 任务特定压缩（优于通用摘要）
- 不是："总结这个文档"
- 而是："总结这个文档中仅与 X 相关的事实，时间范围限定在 Y 之后"
- 保持语义密度的同时显著减少 token 数

### 2. 滑动窗口 + 持久备份
- 最近 N 轮原文保留（高保真推理）
- 旧历史移到持久长期记忆存储
- 关键行动和摘要状态定期"同步"到工作上下文

### 3. 相关性评分
- 使用更小、更快的模型（如 BERT 变体）对检索块评分
- 仅将前 X% 的信息传递给主推理模型

## 结构化上下文注入（Structured Context Injection）

### 原则
用机器可读的格式替代原始非结构化文本，降低 LLM 的认知负担：

1. **Schema 强制注入**：使用严格 JSON 或 XML 格式
2. **分隔符使用**：用标准 XML 标签清晰划分不同类型的上下文
3. **Few-Shot 模板**：用结构化格式注入行为示例

### 为什么重要
- 防止"提示注入"攻击
- 帮助模型区分指令和数据
- 在 agentic 循环中确保一个 agent 的输出能成为另一个的输入

## 反模式（Anti-Patterns）

### 1. 上下文中毒（Context Poisoning）
- **问题**：灌入太多无关数据 → "上下文迷雾" → 推理退化
- **修复**：严格的检索过滤（如重排序）

### 2. 上下文腐烂（Context Rot）
- **问题**：无限增长的对话历史 → 延迟增加 + 信息矛盾
- **修复**：分层记忆 + 上下文裁剪

### 3. "全塞进 Prompt" 谬误
- **问题**：试图用单个"神级 prompt"处理逻辑、检索和工具路由
- **修复**：模块化——将系统指令、工具定义和动态数据分离

### 4. 低效工具/能力面
- **问题**：始终暴露所有可用工具 → token 浪费 + 幻觉函数调用
- **修复**：动态工具路由（基于当前 agent 状态仅暴露相关工具）

## 上下文预算管理

> 如果 agent 在用户说话之前就已达到 >60% 利用率，系统可能太臃肿了。

使用"预算"方法：
- **Compress**：通过摘要或移到长期记忆来缩减旧历史
- **Filter**：仅注入与当前任务高度相关的检索文档
- **Route**：仅加载当前子任务需要的特定工具

## 上下文评估检查清单

每次推理前自问：
1. **必要吗？**（仅包含下一步严格需要的内容）
2. **结构化了吗？**（用 XML/JSON 分离指令和数据）
3. **过时了吗？**（积极裁剪���工具输出或对话历史）
4. **可访问吗？**（数据太大→用"检索"或"查询"工具代替直接放入 prompt）

## 对 evo-skill-creator 的启示

### 直接可应用的

1. **SKILL.md 就是上下文工程产物**：SKILL.md 本质上就是一个"结构化上下文容器"。我们已经在做上下文工程，只是没有用这个术语来思考。

2. **Memory 管理需要分层**：当前 memory/ 目录是扁平结构。应该引入分层策略：
   - 核心 memory（每次必读）← 当前的 learning-plan.md、backlog.md
   - 按需 memory（相关时才读）← knowledge/ 子目录
   - 这正是我们"加载核心 memory → 由智能体自主判断哪些需要读取"的设计思路

3. **SKILL.md 的分层布局优化**：
   - 角色定义和核心规则 → 放在最前面（高注意力区域）
   - 命令定义 → 中间（功能参考区）
   - 数据目录声明、报告格式 → 放在最后面（高注意力区域）

4. **动态工具路由**：不同命令需要不同工具集。learn 需要 WebSearch，go 需要 Write/Edit，review 需要 Read。可以在命令定义中明确每个命令的最小工具集。

5. **上下文预算意识**：
   - 创建的 SKILL.md 不宜过长（控制脆弱性 + 上下文预算双重约束）
   - knowledge/ 文件应有"摘要区"供快速扫描，"详情区"供深入参考
