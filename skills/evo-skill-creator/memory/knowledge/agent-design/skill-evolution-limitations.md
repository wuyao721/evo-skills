# 基于 Skill 的智能体进化——天生缺陷与局限性

> 学习日期：2026-03-26
> 来源：多源综合研究

## 核心结论

基于 Skill（prompt/markdown）的智能体进化是一种**不修改 LLM 参数的外部化进化方式**。它有明确的天花板和无法逾越的局限，但同时也有独特的优势。理解这些边界是设计更好架构的前提。

## 五大天生缺陷

### 1. 上下文窗口——硬性天花板

| 问题 | 说明 |
|------|------|
| 固定上限 | 无论窗口多大（1M token），都是有限的。skill + memory + 对话 + 工具调用共享这个空间 |
| 质量随长度衰减 | 塞入越多 token，模型越倾向浅层模式匹配而非深度推理 |
| "Lost in the Middle" | 模型对中间位置的信息关注度显著低于开头和结尾 |
| 计算成本线性增长 | 上下文越长，延迟和成本越高 |

**对我们的影响**：skill 越多、memory 越丰富，越挤占工作空间。这是 Skill 进化的物理极限。

### 2. 非持久学习——"日志回放"而非"记忆"

| 问题 | 说明 |
|------|------|
| 每次从零开始 | 每次唤醒都是重新读取 markdown 文件，不是真正的"记住" |
| 无法区分重要性 | 无法自动区分持久事实和临时对话 |
| Append-Only 困境 | memory 只增不减，相关性随体积增长而衰减 |
| 无法遗忘 | 没有机制删除或衰减过时信息 |
| 无法变异 | memory 内容不会自动演化，只能被显式改写 |

**对我们的影响**：这是 Skill 进化与真正学习的本质区别。我们的 learn/scan 命令只是"写文件"，不是"学到了"。

### 3. 无法习得新知识——调用已有而非创造未有

| 问题 | 说明 |
|------|------|
| 受限于训练数据 | prompt 无法让模型产生训练数据中不存在的知识 |
| 知识有截止日期 | 模型的世界知识有 cutoff，prompt 无法更新 |
| "调用"而非"学习" | prompt engineering 本质是引导模型调用已有知识，不是教它新东西 |

**对我们的影响**：我们通过 WebSearch 获取新信息写入 memory，但模型理解这些新信息的深度受限于其训练时获得的认知框架。

### 4. 控制脆弱性

| 问题 | 说明 |
|------|------|
| prompt 措辞敏感 | 微小的表述变化可能导致完全不同的行为 |
| 长对话漂移 | 8-12 轮后模型可能偏离初始指令 |
| 与��型偏见竞争 | prompt 指令与模型内在偏好竞争，不一定能赢 |
| 复杂 prompt 自相矛盾 | prompt 越长越容易包含冲突指令 |

**对我们的影响**：SKILL.md 越复杂，行为越不可预测。这是为什么 skill-creator 强调"简洁"的根本原因。

### 5. 错误传播——无法自我纠正

| 问题 | 说明 |
|------|------|
| 早期错误级联 | 推理链中的早期错误会放大到后续每一步 |
| 无检查点 | 没有机制在推理过程中"回退" |
| 信心幻觉 | 模型可能对错误结论表现出高度自信 |

## 三种进化方式对比

| 维度 | Prompt/Skill 进化 | RAG | Fine-tuning |
|------|-------------------|-----|-------------|
| **成本** | 极低 | 中等 | 高 |
| **速度** | 即时 | 分钟级 | 小时~天 |
| **新知识** | 不能（只能通过检索间接获取） | 可以（外部知识库） | 可以（训练数据） |
| **持久性** | 文件持久，理解不持久 | 知识库持久 | 权重持久 |
| **可逆性** | 完全可逆（改回 prompt） | 部分可逆 | 难以可逆 |
| **天花板** | 受模型已有能力限制 | 受检索质量限制 | 受数据和训练限制 |
| **适用场景** | 行为定制、流程编排 | 知识扩展、实时信息 | 深度专业化 |

## 无法逾越的局限（真正的天花板）

1. **不能教模型新的推理能力**——只能用 prompt 引导已有能力的组合
2. **不能突破上下文窗口的物理限制**——无论怎么优化 memory 管理
3. **不能实现真正的"记忆"**——每次唤醒都是从零读取
4. **不能保证行为一致性**——同一 prompt 在不同模型/版本下行为可能不同
5. **天花板随模型架构提升而提升**——prompt 能做到什么取决于底层模型能做到什么

## 对 evo-skill-creator 的启示

### 应该做的
- **拥抱"调用而非学习"的本质**——把 skill 设计为精准的"行为调用器"而非"知识容器"
- **memory 管理是生死线**——必须设计精炼/归档/遗忘机制（backlog #3）
- **保持 SKILL.md 简洁**——复杂 prompt 的控制脆弱性是真实风险
- **用 RAG 补充知识短板**——WebSearch 是我们唯一的"新知识入口"

### 不应该追求的
- 不要试图通过 prompt 让智能体"真正学会"复杂推理
- 不要无限膨胀 memory——越大不等于越好
- 不要期望跨模型/版本的完美一致性
