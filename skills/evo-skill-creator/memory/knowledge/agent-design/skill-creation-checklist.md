# Skill 创建质量检查清单

> 来源：2026-03-26 创建 kavabot-engineer 时的教训——命令定义留空
> 适用于：evo-skill-creator 的 go 命令执行创建流程时

## 创建完成前必须检查

### 1. 所有命令都有完整步骤（不允许留空）

每个命令必须包含：
- **步骤列表**（编号的具体操作步骤）
- **输入**（读什么文件/需要什么信息）
- **产出**（输出什么文件/报告/日志）

**反面示例**（不允许）：
```markdown
### scan — 扫描（广度）
关注技术栈动态。
```

**正面示例**（必须达到的详细度）：
```markdown
### scan — 扫描（广度）
1. 确定扫描范围（Layer 1/2/3）
2. 执行扫描（WebSearch/SSH/读代码）
3. 判断相关性（忽略/加入计划/直接沉淀）
4. 更新关注清单
5. 输出报告 + 追加日志
```

### 2. 八大命令完整性检查

| 命令 | 必须有步骤 | 必须有产出 |
|------|-----------|-----------|
| go | 是 | 任务报告 |
| learn | 是 | 知识文件 + 命令报告 |
| scan | 是 | 关注清单更新 + 命令报告 |
| plan | 是 | 学习计划更新 + 命令报告 |
| review | 是 | 命令报告 |
| suggest | 是 | 命令报告 |
| cron | 是 | 仅日志 |
| status | 是（步骤简单但要列出汇报哪些内容） | 仅日志 |

### 3. 角色特定命令也不能空

如果角色有扩展命令（如 `go add`、`go 排查`），每个都要有完整步骤。

### 4. 数据目录声明检查

SKILL.md 中**必须**包含「数据目录」声明段：

- [ ] 有 `## 数据目录` 段落
- [ ] Memory 路径使用**绝对路径**
- [ ] Output 路径使用**绝对路径**
- [ ] 声明段说明"不受当前工作目录影响"

> 来源：2026-03-27 老板建议——创建时要询问数据目录位置并在 SKILL.md 中清晰声明

### 5. 执行后端与模型路由检查

创建新智能体时，必须明确它未来怎么跑：

- [ ] 已询问用户默认执行后端（`Claude` / `OpenCode` / 混合路由）
- [ ] 如果使用 `OpenCode`，已确认默认模型
- [ ] 如果不同命令需求不同，已明确命令级模型路由
- [ ] 如果角色需要 scheduler，已明确调度配置是否需要按命令覆盖 `executor/model/tools`
- [ ] 没有把“当前可用模型名单/当前免费模型名单”直接写死进 `SKILL.md`
- [ ] `SKILL.md` 只描述获取方式、判断方法和决策流程
- [ ] 动态模型信息已沉淀到 `memory/knowledge/`、`watchlist` 或报告，而不是固化在稳定规则层

> 来源：2026-03-28 老板建议——OpenCode 免费模型应按命令类型路由，scheduler 也应支持命令级覆盖

### 6. 权限策略检查

创建新智能体时，必须明确它”默认能做什么、还需要问什么”：

- [ ] 已询问默认目录访问范围
- [ ] 已询问默认工具权限
- [ ] 已包含**工具 skill 所在目录**的读写权限（`~/.claude/skills/<name>/` 和 `~/.opencode/skills/<name>/`），因为运行时可能从不同工具启动，skill 目录与数据目录可能不同
- [ ] 已区分哪些权限可预授权，哪些必须运行时再次确认
- [ ] 如果角色会接入 scheduler，已考虑无人值守场景下是否会被权限确认卡住
- [ ] 如果涉及不同执行器，已考虑权限实现方式是否存在差异

> 来源：2026-03-28 调度 learn 命令时权限受阻——模型路由之外，还必须设计权限策略
> 补充：2026-03-28 老板指出——权限不仅包含数据目录，还包含工具的 skill 安装目录（不同工具安装位置不同）

### 7. 命名原则检查

Skill 名称必须做到**一看就知道它是什么、做什么**：

- [ ] 名称与所属项目有关联（如 `evo-skills-scheduler` 而非 `tick-daemon`）
- [ ] 名称体现角色/职能本质，而非仅描述技术机制
- [ ] 普通人看到名字就能理解这个 skill 的定位

**反面示例**：`tick-daemon`（只说了"定时调"，没说调什么、为什么调）
**正面示例**：`evo-skills-scheduler`（属于 evo-skills 项目的调度服务）

> 来源：2026-03-27 老板建议——命名核心原则：看到名字就知道它属于哪个项目、做什么事

### 8. 创建流程必须一条龙走到底（不允许中断让用户手动操作）

go 命令创建角色时，步骤 1-11 是**连续流水线**，不是"做完一步问用户要不要继续"。

**规则**：
- [ ] brainstorming 完成后必须**立即**调用 /skill-creator 继续创建，不停下来
- [ ] /skill-creator 完成后必须**立即**注册子智能体 + 输出报告
- [ ] 需要用户决策时**用问题的方式获取**（AskUserQuestion），而不是说"请你手动执行某个命令"
- [ ] 任何步骤都不能以"交给用户去做"结尾——既然知道下一步是什么，就继续做

**反面示例**（不允许）：
```
brainstorming 完成。设计文档已保存。可以交给 /skill-creator 创建完整的 skill 了。
```

**正面示例**（必须达到）：
```
brainstorming 完成。设计文档已保存。现在继续调用 /skill-creator 创建 SKILL.md...
（直接开始创建文件）
```

> 来源：2026-03-28 老板批评——brainstorming 后停下来让用户手动操作是严重的流程中断错误

### 9. Cursor Project Manager 配置同步

创建新角色后，必须将其注册到 Cursor 的 Project Manager 插件配置中：

- [ ] 读取 `memory/private/preferences.md` 中的「Cursor Project Manager 配置同步」段落，获取文件路径和格式
- [ ] 读取配置文件，在数组末尾追加新角色条目
- [ ] 如果配置文件不存在，告知老板后跳过（不阻塞创建流程）

> 来源：2026-04-03 老板指令——所有角色都要在 Cursor Project Manager 中有配置，方便快速切换项目

### 10. private/README.md 必须存在

创建新角色时，必须在 `memory/private/` 下创建 `README.md` 文件：

- [ ] `memory/private/README.md` 存在
- [ ] 包含标准表格（文件名、用途、加载时机）
- [ ] 列出了 private/ 下的所有文件
- [ ] SKILL.md 唤醒流程中包含"加载 private/README.md"步骤

> 来源：2026-04-03 老板建议——强制加载 private/README.md 确保大模型知道何时读取私有数据，避免遗漏或过度加载

### 11. scene-index.md 禁止索引 private/ 文件

scene-index.md 只索引公开数据，私有数据由 private/README.md 管理：

- [ ] `memory/scene-index.md` 中**没有**引用 `memory/private/` 下的任何文件
- [ ] 私有数据的加载时机只在 `memory/private/README.md` 中描述
- [ ] 文件覆盖检查表中不包含 private/ 文件的条目

> 来源：2026-04-03 老板规则——scene-index 只管公开数据，private/README.md 管私有数据，索引层面彻底分离

### 12. SKILL.md 必须精简（< 150 行）

SKILL.md 是 Layer 0（身份+路由），不是所有规则的容器：

- [ ] SKILL.md 总行数 < 150 行
- [ ] 命令段每个命令只有一行简述 + `详见 memory/scenario-sops/cmd-xxx.md` 引用
- [ ] 命令的详细步骤、规则、流程已外置到 `memory/scenario-sops/` 下的 SOP 文件
- [ ] 不包含大段内嵌的工作流程、规则列表、SOP 步骤

> 来源：2026-04-05 老板规则——主入口文件要精简，kavabot-engineer 的重构是标杆

### 13. 分层按需加载（索引驱动）

命令的 SOP 和知识必须通过索引文件按需加载，不能在 SKILL.md 中一次性全部写出：

- [ ] 唤醒流程遵循分层加载顺序：SKILL.md → scene-index.md → private/README.md → scenario-sops/ → knowledge/
- [ ] `memory/scenario-sops/` 目录存在，至少包含 go 命令的 SOP 文件
- [ ] 每个命令的详细 SOP 是独立文件（`cmd-go.md`、`cmd-learn.md` 等）
- [ ] 知识文件按场景按需加载，不在启动时全部加载

> 来源：2026-04-05 老板规则——按场景按需加载索引文件，分层加载
