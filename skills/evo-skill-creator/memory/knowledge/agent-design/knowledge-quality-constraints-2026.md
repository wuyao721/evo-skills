# 知识沉淀质量约束体系（2026）

> 来源：综合分析（RAGAS 框架 + Agent Memory 实践 + 我们自身经验 + tech-solver 反馈）
> 学习时间：2026-04-01（深度学习）
> 类别：agent-design / knowledge-management / quality-assurance
> 波动性：medium（质量维度稳定，具体阈值和策略可能演进）
> 置信度：high（多源交叉验证 + 实践经验支撑）

## 核心命题

**知识沉淀不等于知识积累。没有质量约束的沉淀必然导致知识熵增。**

类比 SlopCodeBench 的发现：代码在迭代中系统性退化。知识库也会退化——冗余堆积、过时不清理、粒度混乱、冲突未解决。

## 质量维度体系

### 已实施的维度

#### 1. 抽象优先原则（已写入 evo-agent-model.md）

```
写入知识前的三层检查：
Layer 1：是否是可复用的抽象（原理/模式/方法论）？
Layer 2：是否包含不必要的具体细节（API 版本号/临时配置/一次性解法）？
Layer 3：如果必须包含细节，是否与抽象分层存放？
```

### 需要新增的维度

#### 2. 过时检测（Staleness Detection）

**问题**：知识文件一旦写入就永远存在，即使其中的信息已经过时。

**方案——YAML 元数据 + 衰减模型**：
```yaml
# 每个知识文件头部应包含
created: 2026-03-28
last_verified: 2026-03-28
volatility: low|medium|high  # 已有
confidence: low|medium|high  # 已有
```

**衰减规则**：
- `volatility: high` → 30 天未验证标记 stale
- `volatility: medium` → 90 天未验证标记 stale
- `volatility: low` → 不自动标记（理论框架不过时）

**review 命令职责**：扫描知识文件，标记 stale 文件，决定更新/归档/删除。

#### 3. 冗余检测（Deduplication）

**问题**：不同学习会话可能沉淀高度重叠的知识。

**方案——学习前查重**：
```
learn 命令在沉淀新知识前：
1. 扫描 knowledge/ 目录下所有文件标题和核心概念
2. 如果新知识与现有文件 >70% 重叠：
   - 合并到现有文件（增量更新）
   - 而非创建新文件
3. 如果新知识是现有知识的深化：
   - 更新现有文件，增加"深化"章节
   - 递增学习次数
```

**当前痛点实例**：学习计划中 #41 出现了两条重复条目（AVO），正是因为不同来源发现了相同主题但未去重。

#### 4. 冲突解决（Conflict Resolution）

**问题**：不同来源可能给出矛盾的结论。

**方案——显式标记冲突**：
```markdown
## 争议点
> ⚠️ 知识冲突：来源A认为X，来源B认为Y
> 当前采信：X（理由：...）
> 待验证：下次 learn 时重新评估
```

**决策优先级**：
1. 有实验数据支撑 > 纯理论推演
2. 多个独立来源一致 > 单一来源
3. 最新研究 > 较早研究（但需注意可复现性）
4. 与我们实践经验一致 > 与实践经验矛盾

#### 5. 粒度控制（Granularity Control）

**问题**：知识文件的粒度不一致——有些是 600 行的综合框架，有些是 30 行的单概念笔记。

**方案——三级粒度标准**：

| 级别 | 适用场景 | 目标长度 | 示例 |
|------|---------|---------|------|
| 概念笔记 | 单个概念/工具/方法 | 30-80 行 | auto-tick-mechanism.md |
| 主题分析 | 一个完整主题的深度分析 | 80-200 行 | evolver-experience-driven-lifecycle-2026.md |
| 综合框架 | 多来源交叉验证的综合框架 | 200-400 行 | constrained-self-evolution-framework-2026.md |

**原则**：超过 400 行 → 拆分为多个文件或提取子主题。

#### 6. 引用溯源（Source Tracing）

**问题**：知识文件的来源有时模糊，无法验证或更新。

**已有的好实践**：
```markdown
> 来源：arXiv:2603.24517（2026-03）
```

**需要强化**：
- 每个关键断言都应可追溯到具体来源
- 综合框架文档中的交叉引用应明确标注哪些结论来自哪个来源
- 区分"来源 A 的原始结论"和"我们的推导/映射"

#### 7. 实践验证标记（Practice Validation）

**问题**：理论知识和实践验证之间缺乏显式关联。

**方案——添加验证状态**：
```markdown
> 实践验证：未验证 | 部分验证 | 已验证
> 验证场景：[描述在哪个子智能体/项目中应用过]
```

## 实施策略

### 立即可做（低成本）
1. learn 命令沉淀前增加查重步骤（检查已有文件标题和概念）
2. 知识文件头部统一添加 `created` 和 `last_verified` 字段
3. 冲突知识显式标记

### 需要框架升级（中成本）
4. review 命令增加"知识库健康扫描"子功能
5. 粒度标准写入 evo-agent-model.md

### 长期演进（高成本）
6. 自动过时检测（结合调度任务定期扫描）
7. 跨智能体知识冲突检测

## 核心抽象

> **知识质量的 7 个维度**：抽象优先、过时检测、冗余检测、冲突解决、粒度控制、引用溯源、实践验证。前 4 个防止知识退化，后 3 个提升知识价值。

> **关键洞察**：知识库和代码库面临相同的退化风险——冗余堆积、结构侵蚀、质量下滑。SlopCodeBench 的教训同样适用于知识管理。
