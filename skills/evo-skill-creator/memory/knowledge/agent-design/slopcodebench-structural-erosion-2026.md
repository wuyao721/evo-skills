# SlopCodeBench — Agent 代码结构性退化与受限自我进化（2026）

> 来源：SlopCodeBench (SCBench), 2026-03-25 发布
> 学习时间：2026-03-30（深度学习）
> 类别：agent-design / quality-assurance
> 波动性：medium（基准测试结果随模型进步可能变化，但退化趋势的结论稳定）
> 置信度：high（11 个模型、20 个问题、93 个检查点的系统性实验）

## 核心发现：Agent 代码在迭代开发中系统性退化

SlopCodeBench 是第一个专门评估 AI Agent 在**长期迭代开发**中代码质量变化的基准测试。核心结论颠覆了"Agent 越用越好"的乐观假设。

## 基准设计

- **20 个问题**，跨 **93 个检查点**
- Agent 必须先构建初始版本，然后**在自己的代码基础上**不断扩展
- **关键设计**：隐藏未来迭代的接口规格和测试用例，迫使 Agent 早期做出架构决策
- **语言无关**，模拟真实世界的软件工程场景

## 两个核心度量

### 1. Verbosity（冗余度）
- 定义：冗余或重复代码的比例
- 结果：**89.8% 的 Agent 轨迹中冗余度单调递增**
- Agent 代码比人类代码冗余 **2.2 倍**

### 2. Structural Erosion（结构侵蚀）
- 定义：复杂度质量集中在高复杂度（膨胀）函数中的比例
- 结果：**80% 的 Agent 轨迹中结构侵蚀单调递增**
- Agent 倾向于在现有函数上"打补丁"而非重构

## 关键实验结果

| 指标 | 数据 |
|------|------|
| 端到端完成率 | **0%**（11 个模型无一完成全部检查点） |
| 最高检查点解决率 | **17.2%**（Opus 4.6） |
| 冗余度增长轨迹比例 | 89.8% |
| 结构侵蚀增长轨迹比例 | 80% |
| Agent vs 人类冗余度对比 | 2.2x |
| Prompt 干预效果 | 改善初始质量，**无法阻止长期退化** |

### "通过率幻觉"

最重要的发现之一：Agent 可以在**功能测试持续通过**的情况下，底层代码架构**快速恶化**。这意味着传统的 pass-rate 基准系统性地**低估了退化问题**。

## 退化根因分析

### 1. 缺乏全局架构意识
- Agent 在每个检查点只看到当前需求，缺乏"项目级"的架构规划能力
- 倾向于"最小改动通过测试"而非"最优架构设计"

### 2. 打补丁而非重构
- Agent 倾向于在现有代码上添加逻辑，而不是识别模式并提取抽象
- 导致函数越来越长、复杂度越来越高

### 3. 无记忆性
- Agent 不记得之前的架构决策，每次迭代都是"新会话"
- 无法回顾和反思之前的设计选择

### 4. Prompt 干预的局限
- "Plan first" 策略可以改善初始代码质量
- 但**无法阻止长期退化率**——退化是系统性的，不是偶发的

## 对自我进化模型的根本影响

### 挑战我们的核心假设

我们假设自我进化智能体会"越来越好"，但 SlopCodeBench 证明：
- **进化不等于改善**：如果没有结构约束，迭代更可能导致退化
- **通过测试不等于质量好**：功能正确但架构腐烂
- **Prompt 指导不够**：需要**机制性保障**，而非仅靠指令

### 受限自我进化（Constrained Self-Evolution）

这是本次学习最重要的新概念：

```
受限自我进化 = 进化能力 + 结构性护栏

进化能力（来自 DGM-H 等框架）：
  - 自我修改、自我改进、跨领域迁移

结构性护栏（来自 SlopCodeBench 启示）：
  - 架构约束（不可违反的结构规则）
  - 退化检测（自动检测冗余度和复杂度增长）
  - 强制重构（达到阈值时必须重构再继续）
  - 版本对比（与基线版本比较质量指标）
```

## 2026 年行业应对策略：Harness Engineering

### 1. 结构确定性原则
- LLM 输出是概率性的，必须强制进入确定性结构
- 机器不可执行的约束 = 不存在的约束

### 2. 三层防护

| 层级 | 机制 | 目标 |
|------|------|------|
| 闭环反馈（Judge） | 自动评估器/测试套件 | 验证结构性要求 |
| 中间件钩子（Hooks） | 权限约束 + 原子变更强制 | 限制修改范围 |
| 环境感知（MCP） | 依赖图 + 架构文档 | 让 Agent 看到边界 |

### 3. Harness First, Agent Second
- 先构建测试套件和"参与规则"，**然后**才构建 Agent
- 如果无法定义"好的改变"的标准，Agent 必然产出"坏的改变"

## 对我们的启示

### 立即可做

1. **review 命令增加"退化检测"**
   - 检查 SKILL.md 的行数增长趋势
   - 检查 knowledge/ 文件的重复度
   - 检查 learning-plan.md 的条目是否过多（表示"追加但不完成"）

2. **evo-agent-model.md 增加"退化防护"章节**
   - 定义结构约束（不可违反的规则）
   - 定义退化阈值（触发强制整合/重构）
   - 定义质量指标（不仅关注功能完成，还关注结构健康）

3. **为子智能体添加"重构触发器"**
   - 当 SKILL.md 超过 N 行 → 触发重构
   - 当 knowledge/ 文件超过 M 个 → 触发整合
   - 当 learning-plan.md 待学习条目超过 K 个 → 触发优先级重排

### 中期规划

1. **自动化质量监控**
   - 定期扫描所有子智能体的"健康指标"
   - 冗余度、复杂度、增长趋势

2. **强制整合机制**
   - learn/scan 沉淀知识后，定期整合相关知识
   - 避免"只追加不整合"的退化模式

### 长期愿景

1. **Harness Engineering 应用到智能体生态**
   - 为每个子智能体定义"结构约束"
   - review 命令充当"Judge"角色
   - 实现"Harness First, Agent Second"原则
