# 技能路由器机制研究

## 概述

随着技能数量增长，依赖description触发的路由方式效率低下，需要行为可训练的技能路由器。

## 关键研究发现

1. **Skill Routing的重要性**：在LLM Agent生态系统中，技能路由是识别和检索最相关技能的关键过程。研究表明，技能的完整实现文本（body field）是真正的区分因素，而不仅仅是名称和描述。

2. **当前局限性**：现有架构通常只呈现技能名称和描述给Agent，而隐藏完整实现。这种方法效果不佳，因为：
   - 移除完整实现文本会使检索准确率下降29-44个百分点
   - 91.7%的注意力集中在body字段上
   - 元数据 alone 不足以进行有效选择

3. **SkillRouter方法**：作为两阶段检索和重排序管道，具有以下特点：
   - 12亿参数（0.6亿编码器 + 0.6亿重排序器）
   - 达到74.0%的top-1路由准确率
   - 在消费级硬件上运行
   - 重新定义了LLM生态系统中的技能选择方式

4. **路由决策框架**：有效的路由器考虑多个标准：
   - **主要信号（权重40%）***: 内容分析（技术术语、财务术语、购买信号等）
   - **次要信号（权重30%）***: 客户上下文（历史记录、最近购买、试用状态等）
   - **紧急信号（权重20%）***: 时间敏感指标（“urgent”、“ASAP”等）
   - **情感信号（权重10%）***: 情感上下文（沮丧语言 vs 中性/积极）

5. **置信度路由**：路由器应该表达对决策的信心：
   - 高置信度 (>80%): 清晰、无歧义的信号，单一明显路由匹配 → 直接路由
   - 中等置信度 (50-80%): 混合信号，多个可能路由 → 路由到最佳匹配，标记审查
   - 低置信度 (<50%): 不清晰或冲突的信号，无强路由匹配 → 路由到人工分队列

## 路由模式变体

1. **分层路由**：对于复杂系统，使用多级路由（域级路由 → 服务类型路由）

2. **语义路由**：使用嵌入进行基于意义的路由：
   - 为每个路由的描述创建嵌入
   - 存储路由嵌入以进行比较
   - 为传入查询生成嵌入
   - 计算与每个路由嵌入的相似度
   - 选择相似度最高且超过阈值的路由

3. **动态路由**：基于当前状态调整路由：
   - 负载路由：如果技术支持队列 > 50，溢出到通用路由
   - 时间路由：营业时间路由到实时代理，非营业时间路由到异步队列
   - 容量路由：在路由前检查处理器可用性，在可用处理器间分配负载

## 实施建议

1. **保持路交叉正交**：路由应该尽可能互斥，以避免冲突
   - 良好示例：按输入类型分路由（图像、文本、音频）
   - 有问题示例：基于重叠特征分路由（包含数字、是问题、较短）

2. **明确记录路由标准**：每个路由应有无歧义的选择标准
   - 使用必需条件和排除条件
   - 明确何时不选择该路由

3. **监控路由分布**：跟踪请求流向以检测路由漂移
   - 每条路由随时间的请求数
   - 路由利用率百分比
   - 溢出频率
   - 质量指标（误路由率、升级率、按路由的解决时间）

4. **版本化路由**：随着路由逻辑演进，保持版本以确保可追溯性
   - 记录版本间的变化
   - 提供迁移说明
   - 现有排队请求使用旧路由，新请求使用新逻辑

## 常见错误

1. **过度复杂路由**：太多路由增加维护难度和混淆
   - 症状：单一级别超过10条路由
   - 解决方案：使用分层路由将相关路由分组

2. **标准不足**：模糊标准导致路由不一致
   - 症状：相同输入在不同运行中路由不同
   - 解决方案：为每条路由添加具体、可衡量的标准

3. **缺少回退**：无默认路由导致失败
   - 症状：某些输入未收到路由决策
   - 解决方案：始终为未匹配输入定义默认路由

4. **忽略上下文**：仅基于内容路由会遗漏重要信号
   - 症状：技术正确但让用户沮丧的路由
   - 解决方案：将用户历史、紧急性和情感纳入路由决策

## 在Evo-Skill-Creator中的应用

在Evo-Skill-Creator中实施技能路由器可以：

1. 改善技能选择的准确性，特别是在技能数量增长时
2. 通过考虑上下文（用户历史、任务类型等）实现更智能的路由
3. 通过置信度评分和回退机制处理不确定性
4. 通过动态路由适应不断变化的工作负载和可用性
5. 支持按需技能加载，仅加载相关技能以避免系统提示过载
