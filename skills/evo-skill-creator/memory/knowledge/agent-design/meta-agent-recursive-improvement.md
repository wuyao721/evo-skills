# Meta-Agent 与递归自我改进

> 2026年3月学习：Agent-Designing-Agents 与 Hyperagents 新趋势

## 核心概念

### Meta-Agent（元智能体）

**定义**：能够设计、创建和改进其他 Agent 的 Agent。

**关键特征**：
1. **递归能力**：不仅能完成任务，还能改进完成任务的方式
2. **自我设计**：能够自动构建和优化 Agent 架构
3. **进化循环**：持续的"改进→评估→再改进"循环

### Hyperagents（超级智能体）

**Meta 的 Hyperagents 核心思想**：
- **"Improve at improving"**：不仅在任务上改进，还在"如何改进"上改进
- **双层优化**：
  - 第一层：优化任务执行能力
  - 第二层：优化优化策略本身

## 与我们的关系

### 我们已经是 Meta-Agent

**evo-skill-creator 本质**：
- 我们创建其他智能体 → Meta-Agent 的核心功能
- 我们通过 review 命令升级子智能体 → 递归改进
- 我们自己也在学习和进化 → 自我改进

**但我们还不是 Hyperagent**：
- 我们的"改进方式"是固定的（learn/scan/review 命令）
- 我们还不能"改进改进方式本身"
- 我们的进化循环是手动触发的，不是自动的

### 差距分析

| 维度 | 我们的现状 | Hyperagent 目标 | 差距 |
|------|-----------|----------------|------|
| 任务执行 | ✅ 创建智能体 | ✅ 创建智能体 | 无 |
| 任务改进 | ✅ review 升级 | ✅ 自动优化 | 手动 vs 自动 |
| 改进策略 | ❌ 固定命令 | ✅ 可进化的策略 | 策略固化 |
| 评估机制 | ❌ 缺失 | ✅ 自动评估 | 无量化指标 |
| 进化触发 | ⚠️ 手动/定时 | ✅ 自适应触发 | 触发机制简单 |

## 实现路径

### 短期（已有基础）

1. **增强评估能力**
   - 为子智能体建立量化指标（连接学习计划 #9 自动技能评估框架）
   - review 命令增加"效果评估"步骤
   - 记录每次升级的效果数据

2. **自动化进化循环**
   - cron 定期触发 review all
   - 基于评估结果自动决定是否升级
   - 建立"升级→评估→反馈"闭环

### 中期（需要新能力）

3. **策略库机制**
   - 将"如何改进"本身抽象为可配置的策略
   - memory/strategies/ 目录存储改进策略
   - review 命令可以选择不同策略

4. **策略进化**
   - 记录每种策略的效果
   - 自动淘汰低效策略
   - 生成新策略（变异现有策略）

### 长期（理论探索）

5. **完全递归自我改进**
   - Agent 能修改自己的 SKILL.md
   - Agent 能创建新的命令
   - Agent 能重新定义自己的目标

## 关键洞察

### 1. 递归的层次

```
Level 0: 执行任务（普通 Agent）
Level 1: 改进任务执行（我们的 review）
Level 2: 改进改进方式（Hyperagent）
Level 3: 改进改进改进方式（理论极限？）
```

### 2. 固定框架 vs 可进化框架

**我们当前的困境**：
- SKILL.md 是固定的 → 命令集固定
- 命令逻辑是固定的 → 改进方式固定
- 这限制了我们的进化上限

**突破方向**：
- 将"命令"本身变成可配置的
- 将"改进策略"变成可学习的
- 但要保持核心约束（目标对齐、安全边界）

### 3. 评估是递归改进的关键

**没有评估 = 盲目进化**：
- 不知道改进是否有效
- 不知道哪种策略更好
- 无法形成正反馈循环

**评估的层次**：
- 任务成功率（Level 0）
- 改进效果（Level 1）
- 策略优劣（Level 2）

### 连接 #9: 自动技能评估框架

- 评估是递归改进的前提
- 没有评估，就无法判断"改进"是否真的改进了
- 需要尽快实现评估机制

### 连接 #16: 自我巩固机制

- 自我巩固是"经验→技能"的转化
- Meta-Agent 是"经验→策略"的转化
- 两者都是从经验中学习，但层次不同
