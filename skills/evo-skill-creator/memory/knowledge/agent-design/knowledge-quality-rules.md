# 沉淀质量准则——避免幻觉污染知识库

> 来源：2026-03-26 自动 tick 研究中的实际错误
> 教训等级：严重

## 事件

研究自动 tick 机制时，WebSearch 返回了 `/loop` 命令、Desktop Scheduling、Cloud Scheduling 等内容。我未经验证就直接写入知识文件，当老板实际尝试 `/loop` 时发现不存在——搜索结果是幻觉或版本不匹配。

## 根因

**搜索结果 ≠ 事实。** WebSearch 返回的内容可能是：
- 模型幻觉（搜索摘要本身是 AI 生成的）
- 过时信息（旧版本有、新版本没了）
- 未发布功能（路线图上有但还没实现）
- 其他产品的功能（张冠李戴）

## 准则

### 沉淀前必须验证

| 内容类型 | 验证方式 | 示例 |
|---------|---------|------|
| CLI 命令/参数 | `--help` 或实际执行 | `claude --help \| grep loop` |
| API/SDK 功能 | 读官方文档或实际调用 | 打开官方文档页面确认 |
| 框架/论文 | 检查原始出处（arXiv/GitHub） | 论文能找到原文就可信 |
| 最佳实践/方法论 | 多源交叉验证 | 至少 2 个独立来源一致 |

### 无法验证时的处理

- **标记为"未验证"**：在知识文件中明确标注 `> ⚠️ 未验证：以下内容来自搜索结果，未经实际测试`
- **不要混入已验证内容**：未验证和已验证的内容要分开，不要混在一起给人已确认的错觉
- **优先级降低**：未验证的知识不应直接用于指导行动，只能作为"待验证线索"

### 错误处理

发现已沉淀的知识有误时：
1. **立即修正**知识文件（不是追加修正说明，而是重写正确内容）
2. **记录教训**到本文件
3. **检查是否有其他地方引用了错误知识**（backlog、SKILL.md、evo-agent-model.md）
4. 同步修正所有引用

## 事件二：把动态事实错误写入 SKILL.md

### 事件

在沉淀 OpenCode 免费模型路由时，我把当前观察到的模型名单直接写进了 `SKILL.md` 的创建流程，例如：

- `opencode/minimax-m2.5-free`
- `opencode/mimo-v2-pro-free`
- `opencode/nemotron-3-super-free`

老板指出这是错误的，因为这类信息会变化，而且不同用户本地执行 `opencode models` 得到的结果也可能不同。

### 根因

**我混淆了“稳定规则”和“动态事实”的边界。**

- `SKILL.md` 应保存稳定的方法、提问流程、决策框架
- memory/knowledge 应保存可更新的知识总结
- watchlist/report 应保存阶段性发现和变化线索

我这次的问题不是“推荐思路完全错误”，而是**把本应动态维护的信息写进了必须长期稳定的文件**。

### 新准则：稳定规则与动态事实分层

#### 一、`SKILL.md` 只能放稳定内容

允许放入 `SKILL.md` 的内容：

1. 提问流程
2. 决策维度
3. 方法论
4. 固定目录与产出规则
5. 哪类信息应该去哪里获取

不应放入 `SKILL.md` 的内容：

1. 当前可用模型名单
2. 当前免费模型名单
3. 当前版本号
4. 当前 CLI 输出结果
5. 任何会因环境、账号、时间变化而失效的事实

#### 二、动态事实必须进入可更新层

动态事实应优先放在这些地方：

1. `memory/knowledge/`：相对稳定但可更新的知识总结
2. `memory/watchlist.md`：扫描发现但尚未确认或尚未学透的变化
3. `output/report/*.md`：当次观察、对比、建议
4. 本地命令输出：如 `opencode models`

#### 三、写入前必须问自己两个问题

在把信息写入 `SKILL.md` 或通用模型前，先检查：

1. 这条信息是否会因用户环境不同而变化？
2. 这条信息是否会因时间推移而过期？

只要任一答案是“会”，就**不能直接写进 `SKILL.md`**，而应改写成：

- “通过什么方式获取”
- “获取后如何判断”
- “判断结果沉淀到哪里”

### 教训升级

第一类错误是“未经验证就沉淀事实”。
第二类错误是“事实即使真实，也可能写错层级”。

所以完整规则应是：

1. **先验证事实是否真实**
2. **再判断事实应该沉淀在哪一层**

真实，不代表适合写进 `SKILL.md`。
