# 执行器权限策略设计

> 沉淀日期：2026-03-28
> 来源：scheduler 自动执行 `evo-skill-creator learn` 时触发权限阻塞

## 事件

在通过 `evo-skills-scheduler` 自动执行 `evo-skill-creator learn` 时，出现了权限请求。

现象：

1. 调度器当前工作目录在 `scheduler/`
2. skill 的数据目录位于 `~/.claude/skills/evo-skill-creator/`
3. 执行器需要访问 skill 自己的 `references/`、`memory/`、`output/`
4. 由于这些目录不在默认可访问范围内，任务在无人值守场景下被权限确认打断

## 结论

权限策略必须成为“创建角色”时的显式设计项，而不是运行时临时补救项。

## 权限策略应包含什么

### 1. 默认目录访问范围

至少要明确：

1. 角色自己的 `memory/`
2. 角色自己的 `output/`
3. 角色自己的 `references/`
4. 相关项目代码目录
5. `scheduler/configs/` 等配套配置目录（如需要）

### 2. 默认工具权限

至少要明确：

1. 哪些工具默认允许
2. 哪些工具默认禁止
3. 哪些高风险工具必须额外确认

### 3. 权限申请机制

要区分两类：

1. **预授权**：创建角色时就明确授予，适合高频、稳定、低争议的访问
2. **运行时申请**：只有在偶发或高风险场景才临时询问老板

## 当前已确认的执行器差异

### Claude

从 `claude --help` 可确认，当前 CLI 暴露了以下相关能力：

1. `--add-dir`：补充可访问目录
2. `--allowedTools` / `--disallowedTools`：控制工具权限
3. `--permission-mode`：控制权限询问模式
4. `--dangerously-skip-permissions`：跳过权限检查（仅适合安全环境）

说明：Claude 的权限策略可以比较明确地落到命令行参数层。

### OpenCode

从 `opencode run --help` 可确认，当前 CLI 暴露了：

1. `--dir`：指定运行目录
2. `--agent` / `-m` 等执行相关参数

但当前帮助信息里**没有像 Claude 一样直接暴露工具白名单/权限模式参数**。

说明：OpenCode 的“权限策略”在创建角色时仍要设计，但具体落地方式可能与 Claude 不同，不能简单假设两者有相同的 CLI 参数。

## 对角色创建流程的要求

创建新角色时，必须主动问用户两类问题：

1. **默认权限**：这个角色默认应该拥有哪些目录和工具权限？
2. **额外权限**：除了默认权限，还需要哪些可选权限？它们是在创建时一次性授予，还是运行时再问？

推荐问法：

1. 这个角色默认需要访问哪些目录？
2. 这些目录里，哪些是高频刚需，应该预授权？
3. 这个角色默认需要哪些工具？
4. 是否允许 scheduler 场景下沿用同一套权限？
5. 哪些额外权限必须每次单独询问老板？

## 对 scheduler 的启示

调度配置不应只管理 `executor/model/tools`，还应考虑：

1. 默认目录访问范围
2. 权限模式
3. 命令级权限覆盖

理想结构：

1. `agent` 级默认：`executor/model/tools/permissions`
2. `schedule` 级覆盖：某个命令单独指定 `executor/model/tools/permissions`

## 一句话结论

**模型路由决定“谁来做”，权限策略决定“能不能做”。**

如果只设计模型，不设计权限，scheduler 在真实自动化场景里会经常卡在运行时确认上。
